## 1. Общие сведения

### 1.1. Назначение и контекст

**ouroboros** — микросервис автономного самоулучшающегося AI-агента. Это форк, предназначенный для развёртывания в виде облачного микросервиса (Kubernetes, режим 24/7 always-on) с проприетарными коннекторами и интеграциями.

Сервис решает задачу размещения автономного агента, который:

- ведёт диалог с человеком через Web UI (чат, онбординг, настройки);
- исполняет задачи и инструменты (tools), подключаемые навыки (skills) и расширения (extensions);
- работает в автономном цикле (agent loop) — самоанализ, рефлексия, ведение backlog улучшений, пред-коммитное ревью собственных изменений;
- взаимодействует с другими агентами по протоколу A2A (agent-to-agent);
- синхронизирует собственный код с git-репозиторием (auto-sync).

Архитектурно сервис — единый Python-процесс (ASGI, Starlette + Uvicorn): HTTP/WebSocket-фасад (`server.py`), доменное ядро (пакет `ouroboros/`), супервизор воркеров и шина сообщений (`supervisor/`), статика Web UI (`web/`). Внешней реляционной БД нет — состояние хранится в файловой плоскости данных (JSON/JSONL).

Потребители сервиса:

- **Web-пользователь** — человек, взаимодействующий через браузер (чат, онбординг, управление настройками/навыками);
- **A2A-агент** — внешний агент, отправляющий задачи по протоколу A2A;
- **Платформа / автоматика** — supervisor, agent-loop-tick, фоновое «сознание», git auto-sync (триггеры без участия человека);
- **Админ / оператор** — управление конфигурацией, runtime-режимом, marketplace, локальными моделями, мониторинг.

### 1.2. Глоссарий доменных терминов

| Термин | Синонимы | Определение |
|--------|----------|-------------|
| Агент | Agent, Ouroboros | Автономный исполнитель: ведёт диалог, исполняет задачи, в автономном цикле модифицирует собственный код в пределах разрешённого runtime-режима |
| Задача | Task, TaskResult | Единица работы агента с уникальным `task_id` и техническим статусом. Результат и статус персистятся в файл `task_results/{task_id}.json` |
| Воркер | Worker | Процесс-исполнитель задачи, порождаемый супервизором. Число ограничено `OUROBOROS_MAX_WORKERS` |
| Супервизор | Supervisor | Подсистема управления воркерами и шиной сообщений (`supervisor/`), очередь pending/running, учёт бюджета |
| Шина сообщений | Message Bus | Внутренний транспорт между супервизором, воркерами и WebSocket-фасадом (`supervisor/message_bus.py`) |
| Agent loop | Цикл агента | Автономный цикл рассуждение→инструмент→рефлексия (`loop*.py`) |
| Фоновое сознание | Background Consciousness | Фоновый цикл с интервалом пробуждения (wakeup backoff), инъекция наблюдений и отложенных событий (`consciousness.py`) |
| Навык | Skill, SkillManifest | Подключаемая единица возможностей (instruction / script / extension), описанная манифестом `SkillManifest` (схема v1) |
| Расширение | Extension | Навык типа `extension`: добавляет HTTP-маршруты и/или WebSocket-обработчики в runtime-реестр |
| Память | Memory | Долгоживущее состояние агента: scratchpad-блоки, identity, журналы (`memory.py`) |
| Backlog улучшений | Improvement Backlog | Список самостоятельно сформированных улучшений с дедупликацией по fingerprint (`improvement_backlog.py`) |
| Ревью-триада | Review triad | Пред-коммитное ревью изменений тремя LLM-моделями из `OUROBOROS_REVIEW_MODELS`; режим применения — `advisory` (рекомендательный) или `blocking` (блокирует коммит при провале) |
| Scope-review | Ревью охвата | Отдельная проверка соответствия изменений заявленному охвату задачи (не превышает ли агент рамки поручения), выполняется в дополнение к ревью-триаде |
| Runtime-режим | Runtime mode | `light` / `advanced` / `pro` — область разрешённой модификации кода. Owner-only, защищён от эскалации |
| WEBUI_ONLY | — | Конфигурационный тогл (bool). Текущая семантика: отключает все каналы коммуникации кроме Web UI (на текущую дату фактически гейтит Telegram-мост). Подлежит переопределению под проприетарные коннекторы (см. 4.6) |
| Защищённая поверхность | Protected surface | Набор файлов, запись в которые ограничена runtime-режимом (BIBLE.md, safety.py, runtime_mode_policy.py, tools/registry.py, ci.yml, Dockerfile) |
| Tools whitelist | Белый список инструментов | `OUROBOROS_TOOLS_ENABLED` — список разрешённых инструментов; пустой = все авто-обнаруженные |
| A2A | Agent-to-Agent | Протокол межагентного взаимодействия (сервер + executor + файловое хранилище задач `FileTaskStore`) |
| Marketplace | ClawHub / OuroborosHub | Внешние реестры навыков для поиска и установки |
| Git auto-sync | Автосинхронизация | Автоматическая синхронизация репозитория агента с удалённым git (bootstrap при старте, push при graceful shutdown) |
| Owner mailbox | Почтовый ящик владельца | Пер-задачный канал инъекции сообщений владельца воркеру (`memory/owner_mailbox/{task_id}.jsonl`) |
| Плоскость данных | Data plane, DATA_DIR | Корневой каталог файлового состояния (логи, state, навыки, settings.json) |

### 1.3. Бизнес-инварианты

1. **ВСЕГДА** инструмент, отсутствующий в белом списке `OUROBOROS_TOOLS_ENABLED` (при непустом списке) и не входящий в защищённое ядро, недоступен агенту.
2. **НИКОГДА** запись в файлы защищённой поверхности не выполняется в runtime-режиме, который её не разрешает (`light` блокирует запись в репозиторий; `advanced` — эволюционный слой; `pro` — защищённая поверхность только после ревью).
3. **НИКОГДА** значение `OUROBOROS_RUNTIME_MODE` не повышается через HTTP POST `/api/settings`; сравнение идёт с baseline `OUROBOROS_BOOT_RUNTIME_MODE`, зафиксированным на старте и проброшенным в дочерние процессы.
4. **ВСЕГДА** секретные параметры (`*_API_KEY`, `*_TOKEN`, `*_PASSWORD`) в ответе `GET /api/settings` маскируются (`***` или `***`+суффикс).
5. **НИКОГДА** git auto-sync не выполняет операций, ведущих к потере коммитов (force-push не используется); shutdown-push ограничен бюджетом ожидания 25 с.
6. **ВСЕГДА** запись файлов состояния (TaskResult, A2A-задачи, ReviewState) выполняется атомарно по схеме «временный файл → rename».
7. **ВСЕГДА** доступ с не-loopback адреса требует валидного сетевого пароля (`OUROBOROS_NETWORK_PASSWORD`, HMAC-SHA256); loopback (127.0.0.1, ::1, localhost) проходит без пароля.
8. **НИКОГДА** запись в `OUROBOROS_RUNTIME_MODE` и иные owner-only ключи не выполняется через сетевой merge настроек.

> Инвариант «секреты не хранятся в env / конфигурации» в текущем коде **не выполняется** (секреты читаются из переменных окружения и `settings.json`) — см. ограничение 4.6.

#### 2.1.2. Обработчики по триггерам

##### 2.1.2.1. Обработчик: GET /api/health

- **Триггер**: HTTP GET /api/health
- **Инициатор**: автоматика (k8s readiness probe), админ
- **Вход**: нет
- **Предусловия (guard)**: нет (публичный путь, без сетевой аутентификации)

**Шаги обработки:**

1. Сформировать ответ со статусом и версиями (`status`, `runtime_version`, `app_version`).
2. Вернуть 200 + JSON.

- **Side-effects**: нет.
- **Выход / ответ**: 200 OK + `HealthResponse`.
- **Ошибочные ветки**: отсутствуют (эндпоинт легковесный, без зависимостей).
- **Идемпотентность**: да, по природе (read-only GET).

##### 2.1.2.2. Обработчик: GET /api/state

- **Триггер**: HTTP GET /api/state
- **Инициатор**: Web-пользователь, админ, автоматика (liveness)
- **Вход**: нет
- **Предусловия (guard)**: сетевая аутентификация (кроме loopback) — см. 4.1
- **Шаги обработки:**
  1. Собрать снапшот: `uptime` (от `APP_START`), `workers_alive`/`workers_total`, `pending_count`, `running_count`, `spent_usd`, `budget_limit`, `budget_pct`, `evolution_cycle`, `supervisor_ready`, `supervisor_error`, `runtime_mode`, `skills_repo_configured`, история чата.
  2. Вернуть 200 + JSON.
- **Side-effects**: нет.
- **Выход / ответ**: 200 OK + `StateResponse`.
- **Ошибочные ветки**: 401/403 при отсутствии сетевой аутентификации.
- **Идемпотентность**: да (read-only).

##### 2.1.2.3. Обработчик: GET/POST /api/settings

- **Триггер**: HTTP GET /api/settings (чтение), POST /api/settings (изменение)
- **Инициатор**: админ / оператор
- **Вход**: для POST — частичный payload настроек (JSON)
- **Предусловия (guard)**: сетевая аутентификация (кроме loopback)
- **Шаги обработки:**
  1. GET: загрузить настройки, **замаскировать секреты** (`_SECRET_SETTING_KEYS`), вернуть с блоком `_meta`.
  2. POST: получить файловый lock на `settings.json` (таймаут 2 с).
  3. Смержить payload в текущие настройки; **исключить owner-only ключи** (`OUROBOROS_RUNTIME_MODE` и др.) из сетевого merge (инвариант 1.3.8).
  4. Проверка анти-эскалации runtime-режима относительно `OUROBOROS_BOOT_RUNTIME_MODE` (инвариант 1.3.3, правило 2.3.3).
  5. Сохранить атомарно; пометить ключи как hot-reload или restart-required (см. 2.8).
- **Side-effects**: запись `settings.json` (atomic), частичный hot-reload.
- **Выход / ответ**: 200 OK + применённые настройки (маскированные).
- **Ошибочные ветки**: таймаут lock → ошибка; попытка изменить owner-only ключ — игнорируется/отклоняется (см. 2.2.2).
- **Идемпотентность**: повторный одинаковый payload не меняет состояние; ключа дедупликации запроса нет.

##### 2.1.2.4. Обработчик: WebSocket /ws

- **Триггер**: WebSocket /ws
- **Инициатор**: Web-пользователь (браузер)
- **Вход**: фреймы `ChatInbound` (сообщение) / `CommandInbound` (команда); см. 2.7
- **Предусловия (guard)**: сетевая аутентификация (кроме loopback); соединение установлено
- **Шаги обработки:**
  1. Принять входящий фрейм, определить `type`.
  2. `ChatInbound` → передать в супервизор/шину сообщений как пользовательское сообщение/задачу. Исполнение инструментов в рамках порождённой задачи подчиняется политике инструментов и runtime-режиму (см. правило 2.3.6, 2.3.2).
  3. `CommandInbound` → выполнить команду (`cmd`); при недоступности шины — команда поглощается (см. 2.2.3).
  4. Диспетчеризовать на extension-обработчик, если `type` зарегистрирован в runtime extension registry (permission `ws_handler`).
  5. Транслировать исходящие фреймы (`ChatOutbound`/`LogOutbound`/`TypingOutbound`/`PhotoOutbound`) через `broadcast_ws`.
- **Side-effects**: публикация в шину сообщений; рассылка всем подключённым клиентам; при ошибке отправки клиенту — `broadcast_partial_failure` (см. 4.5).
- **Выход / ответ**: поток исходящих WS-фреймов.
- **Ошибочные ветки**: «мёртвые» клиенты удаляются при исключении отправки; heartbeat-loop детектит живость.
- **Идемпотентность**: не идемпотентно; дедупликация на уровне `client_message_id` / lifecycle `dedupe_key` (где применимо), отдельного request-ключа нет.

##### 2.1.2.5. Обработчик: A2A — приём задачи

- **Триггер**: A2A-протокол (send task) на A2A-сервере (`A2A_HOST:A2A_PORT`)
- **Инициатор**: внешний A2A-агент
- **Вход**: A2A-задача (Pydantic-модель `Task` из `a2a.types`)
- **Предусловия (guard)**: `A2A_ENABLED=true` (иначе сервер не поднят); число активных задач < `A2A_MAX_CONCURRENT`
- **Шаги обработки:**
  1. Принять задачу, сохранить в `FileTaskStore` (`a2a_tasks/{task_id}.json`, atomic).
  2. `OuroborosA2AExecutor` диспетчеризует задачу в супервизор (`_dispatch_to_supervisor`).
  3. Ожидать событие ответа (`response_event.wait`) с таймаутом `OUROBOROS_HARD_TIMEOUT_SEC` (по умолч. 1800 с).
  4. По завершении — обновить статус задачи (`completed`/`failed`/`canceled`/`rejected`).
  5. TTL-очистка задач старше `A2A_TASK_TTL_HOURS` (24 ч).
- **Side-effects**: файлы `a2a_tasks/*.json`; диспетчеризация в супервизор.
- **Выход / ответ**: статус задачи по A2A-протоколу.
- **Ошибочные ветки**: таймаут → `TimeoutError` (без retry); см. 2.2.4.
- **Идемпотентность**: явного request-dedup ключа нет; уникальность по `task_id`; TTL-очистка предполагает уникальность.

##### 2.1.2.6. Обработчик: git auto-sync (bootstrap / shutdown)

- **Триггер**: lifespan startup (bootstrap), lifespan shutdown (push)
- **Инициатор**: сам сервис
- **Вход**: конфигурация git-remote, GitHub-токен / git-пароль
- **Предусловия (guard)**: настроен remote; доступен токен
- **Шаги обработки:**
  1. Startup: `bootstrap_remote_sync` — проверка/инициализация связи с remote (async, без явного retry/backoff).
  2. Shutdown: `shutdown_push_sync` — финальный push, ограниченный `Event.wait` бюджетом 25 с (не превышает grace-период).
- **Side-effects**: git fetch/pull/push (subprocess + HTTPS).

### 2.2. Альтернативные и ошибочные сценарии

#### 2.2.1. <font color="red">**Alt**</font> Доступ без сетевого пароля (не-loopback)

1. Запрос приходит с не-loopback адреса без валидного `ouroboros-auth`/cookie.
2. `NetworkAuthGate` отклоняет запрос (кроме публичных путей `/api/health`, `/auth/login`, `/auth/logout`) — см. правило 2.3.4.
3. Клиент получает отказ авторизации; должен пройти `POST /auth/login`.

#### 2.2.2. <font color="red">**Alt**</font> Попытка изменить owner-only ключ через сеть

1. В `POST /api/settings` передан `OUROBOROS_RUNTIME_MODE` (или иной owner-only ключ).
2. Ключ исключается из merge (инвариант 1.3.8); анти-эскалация сверяется с `OUROBOROS_BOOT_RUNTIME_MODE` (правило 2.3.3).
3. Значение не изменяется; остальные допустимые ключи применяются.

#### 2.2.3. <font color="blue">**Opt**</font> Шина сообщений недоступна при WS-команде

1. При обработке `CommandInbound` супервизор/`message_bus` недоступен.
2. Команда поглощается (swallowed), состояние не меняется (риск потери команды — см. ограничение 4.6).

#### 2.2.4. <font color="red">**Alt**</font> Таймаут A2A-задачи

1. `OuroborosA2AExecutor` ожидает ответ дольше `OUROBOROS_HARD_TIMEOUT_SEC`.
2. Поднимается `TimeoutError`, retry не выполняется (см. правило 2.3.5).
3. Задача завершается со статусом ошибки; запись остаётся до TTL-очистки.

#### 2.2.5. <font color="red">**Alt**</font> Запрещённая запись в защищённую поверхность

1. Агент пытается записать файл из защищённой поверхности в неразрешённом runtime-режиме.
2. Срабатывает sandbox реестра инструментов / `runtime_mode_policy` (правило 2.3.2), запись блокируется или ревертится (`claude_code_edit` revert guard).
3. Действие отклонено; инвариант 1.3.2 сохранён.

### 2.3. Бизнес-правила и логика обработки

1. **Маскирование секретов в ответе настроек.**
   - **Формулировка:** **ЕСЛИ** ключ настройки входит в `_SECRET_SETTING_KEYS`, **ТОГДА** в ответе значение заменяется маской, **ИНАЧЕ** возвращается как есть.
   - **Псевдокод:**
     1. для каждого ключа K в настройках:
     2. если K в множестве секретных ключей:
     3. v = значение; если длина(v) > 8 → вернуть первые 8 символов + "..."; иначе вернуть "***";
     4. иначе вернуть значение без изменений.
   - **Края:** пустое значение → "***"; отсутствующий ключ не добавляется.

2. **Контроль записи по runtime-режиму.**
   - **Формулировка:** **ЕСЛИ** целевой путь относится к защищённой поверхности, **ТОГДА** запись разрешена только если текущий режим её допускает (`light`: запрет записи в репозиторий; `advanced`: эволюционный слой; `pro`: защищённая поверхность только после прохождения ревью), **ИНАЧЕ** запись отклоняется/ревертится.
   - **Псевдокод:**
     1. категория = классифицировать_путь(target);
     2. если категория ∈ защищённая поверхность:
     3. если НЕ mode_allows_protected_write(текущий_режим, категория) → отклонить;
     4. иначе если требуется ревью → допускать только после успешного ревью;
     5. иначе → разрешить.
   - **Края:** регистронезависимое сравнение путей; неизвестный путь трактуется как непривилегированный.

3. **Анти-эскалация runtime-режима.**
   - **Формулировка:** **ЕСЛИ** запрошенный режим «выше» baseline (`OUROBOROS_BOOT_RUNTIME_MODE`), **ТОГДА** изменение отклоняется; **ИНАЧЕ** допускается только из owner-контекста (не из сетевого merge).
   - **Псевдокод:**
     1. baseline = OUROBOROS_BOOT_RUNTIME_MODE (зафиксирован на старте, проброшен в дочерние процессы);
     2. если порядок(requested) > порядок(baseline) → отклонить (нельзя повысить);
     3. если источник = сетевой merge → отклонить (owner-only);
     4. иначе → применить.
   - **Края:** порядок режимов light < advanced < pro; отсутствующее значение → baseline.

4. **Сетевая аутентификация (loopback bypass).**
   - **Формулировка:** **ЕСЛИ** адрес источника loopback (127.0.0.1, ::1, localhost, IPv6-loopback) **ИЛИ** путь публичный (`/api/health`, `/auth/login`, `/auth/logout`), **ТОГДА** пропустить без пароля; **ИНАЧЕ** требовать валидный HMAC-SHA256 токен в заголовке `ouroboros-auth` или cookie `ouroboros_auth`.
   - **Псевдокод:**
     1. если is_loopback(remote) → allow;
     2. если path ∈ public_paths → allow;
     3. token = header || cookie; если token отсутствует → deny;
     4. если HMAC_valid(token, OUROBOROS_NETWORK_PASSWORD) → allow; иначе deny.
   - **Края:** пустой `OUROBOROS_NETWORK_PASSWORD` при не-loopback доступе — см. ограничение 4.6.

5. **Исполнение A2A-задачи с жёстким таймаутом.**
   - **Формулировка:** **ЕСЛИ** ответ не получен за `OUROBOROS_HARD_TIMEOUT_SEC`, **ТОГДА** поднять `TimeoutError` без повторов; **ИНАЧЕ** вернуть результат и обновить статус.
   - **Псевдокод:**
     1. сохранить задачу в FileTaskStore (atomic);
     2. диспетчеризовать в супервизор;
     3. ждать response_event с таймаутом hard_timeout;
     4. при таймауте → статус=failed (без retry); иначе → статус из результата;
     5. periodic TTL-cleanup задач старше A2A_TASK_TTL_HOURS.
   - **Края:** параллелизм ограничен `A2A_MAX_CONCURRENT`; нет идемпотентного ключа запроса.

6. **Политика инструментов (tool safety).**
   - **Формулировка:** видимость и допустимость инструмента определяются слоями: (1) hardcoded sandbox реестра (блок защищённых путей и мутирующего git); (2) `POLICY_SKIP` для доверенных core/meta-инструментов; (3) `POLICY_CHECK_CONDITIONAL` для `run_shell` (whitelist безопасных субъектов); (4) `POLICY_CHECK` для неизвестных инструментов (проверка дешёвой LLM); поверх — белый список `OUROBOROS_TOOLS_ENABLED`.
   - **Псевдокод:**
     1. если инструмент нарушает sandbox реестра → запрет;
     2. если whitelist непуст И инструмент ∉ whitelist И ∉ защищённое ядро → скрыт (инвариант 1.3.1);
     3. если инструмент ∈ core/meta → POLICY_SKIP (разрешить);
     4. если инструмент = run_shell → POLICY_CHECK_CONDITIONAL по whitelist субъектов;
     5. иначе → POLICY_CHECK (LLM-проверка) → разрешить/запретить.
   - **Края:** формат whitelist (разделитель/экранирование) в коде не специфицирован — см. ограничение 4.6.

### 2.4. Состояния и переходы (FSM)

**Не применимо для данного сервиса.** Строгого доменного FSM с бизнес-переходами у сервиса нет.

В коде присутствуют **технические статусы** (не доменный FSM):

- `TaskResult.status` ∈ `{requested, scheduled, running, completed, rejected_duplicate, failed, interrupted, cancelled}` — отражает технический жизненный цикл исполнения задачи (см. модель 2.7 `TaskResult`);
- A2A-задача: `status.state` ∈ `{completed, failed, canceled, rejected}`;
- `AdvisoryReviewState` — внутренняя машина состояния пред-коммитного ревью (runs/attempts/obligations/debts), деталь реализации, не доменный контракт.

`TaskResult.status` — **технический жизненный цикл исполнения** (внутренняя реализационная деталь конвейера задач), а не доменная FSM: переходы не несут бизнес-guard-логики, не управляются бизнес-правилами из 2.3 и не являются частью внешнего контракта (A2A-клиент наблюдает лишь терминальное `status.state`). Поэтому эти статусы документируются исключительно как атрибуты моделей данных (2.7) и НЕ выносятся в отдельную доменную FSM.

### 2.6. Интеграции

#### LLM-провайдеры (исходящие, HTTP)

- **Anthropic API** — клиент Anthropic, базовый `https://api.anthropic.com/v1`, методы `POST /messages`, `POST /text_completions`. Ключ `ANTHROPIC_API_KEY`. Таймаут запроса в коде явно не задан (дефолт SDK) — зона доработки, см. 4.6.
- **OpenRouter** — `https://openrouter.ai/api/v1`, `GET /models`, `POST /chat/completions`. Таймаут запроса capabilities — 15000 мс (захардкожен). Заголовок `HTTP-Referer: https://ouroboros.local/`.
- **Локальная LLM (llama-cpp-python)** — `POST /v1/chat/completions`, `POST /v1/embeddings`; порт/автозапуск подпроцесса локальной модели.
- Слой абстракции провайдеров (единая точка маршрутизации LLM-вызовов) — точка подключения корпоративного LLM-шлюза; форк-специфика, см. 4.6.

#### A2A-сервер (входящий)

- **Транспорт**: A2A-протокол, `A2A_HOST:A2A_PORT` (по умолчанию `127.0.0.1:18800`), включается `A2A_ENABLED` (restart-required).
- **Хранилище задач**: `FileTaskStore` (`a2a_tasks/{task_id}.json`, atomic, TTL `A2A_TASK_TTL_HOURS`=24 ч).
- **Исполнение**: `OuroborosA2AExecutor` → диспетчеризация в супервизор; таймаут `OUROBOROS_HARD_TIMEOUT_SEC`; параллелизм `A2A_MAX_CONCURRENT`=3; retry нет.

#### Git auto-sync (исходящий)

- **Транспорт**: subprocess `git` + HTTPS к remote (`https://<host>/<org>/<repo>.git`).
- **Аутентификация**: GitHub-токен (`x-access-token` / PAT) либо `OUROBOROS_GIT_PASSWORD`.
- **Поведение**: bootstrap при старте (async, без явного retry/backoff); push при graceful shutdown (бюджет 25 с); force-push не используется (инвариант 1.3.5). Предполагается ветка по умолчанию (мульти-ветка не поддержана — см. 4.6).

#### Marketplace (исходящий, HTTP)

- **ClawHub** — `GET /search`, `/info`, `/download`, `/list-versions`; rate-limit retry до 5 раз, max sleep 60 с (хардкод).
- **OuroborosHub** — `GET /catalog.json`.

#### Claude Code gateway

- **Назначение**: исполнение Bash / чтение файлов / правка кода через Claude Code Runtime; применяются guard'ы пути и read-only-режима (ограничивают доступ к файловой системе по политике, см. правило 2.3.2/2.3.6); таймаут — task-scoped.

#### Шина сообщений (внутренняя)

- `supervisor/message_bus.py` — транспорт супервизор ↔ воркеры ↔ WS-фасад. При недоступности команда WS поглощается (см. 2.2.3).

> Kafka, gRPC, SOAP, реляционная БД и cron-планировщик — **не используются**. Headless-браузер (Playwright/Chromium) как исходящая runtime-интеграция — **не используется**: browser-инструментарий агента (`browse_page`/`browser_action`) и веб-fetch (`web_fetch`/`fetch`) удалены, runtime-зависимости `playwright`/`playwright-stealth` сняты (см. 4.6).

### 2.11. Обработка ошибок

Единый стандарт ошибок (RFC 7807 / машинный `errorCode`) в сервисе **отсутствует**. Фактическое поведение:

| HTTP-код | errorCode | Ситуация | Действие клиента |
|----------|-----------|----------|-----------------|
| 200 | — | Успех (включая часть ошибочных бизнес-кейсов с телом `{"error": "..."}`) | Разобрать поле `error` |
| 401/403 | — | Нет валидной сетевой аутентификации (не-loopback) | Выполнить `POST /auth/login` |
| 4xx | — | Невалидный ввод (формат тела/пути) | Исправить запрос |
| 5xx | — | Внутренняя ошибка | Повторить позже |

Типовой формат ошибки (фактический, нестандартизованный):
```json
{ "error": "human-readable message" }
```

Известные исключения исполнения: `ClawHubClientError`, `ClawHubRateLimitError` (marketplace), `TimeoutError` (A2A). Стандартизация кодов ошибок — зона доработки (см. 4.6).

## 4. Нефункциональные требования

### 4.1. Безопасность

Стандартный набор платформы (целевые требования; фактическое состояние и расхождения — в 4.6):

1. Взаимодействие между сервисами ДОЛЖНО осуществляться по TLS 1.2+.
2. Для межсервисного взаимодействия ДОЛЖЕН использоваться mTLS.
3. Управление доступом ДОЛЖНО осуществляться на основе ACL.
4. Секреты НЕ ДОЛЖНЫ храниться в коде, конфигурации, переменных окружения.
5. Конфиденциальная информация ДОЛЖНА маскироваться в логах на 100%.

Фактическое поведение сервиса:

6. Сетевой доступ контролируется `NetworkAuthGate`: loopback — без пароля; не-loopback — HMAC-SHA256 пароль (`OUROBOROS_NETWORK_PASSWORD`) в заголовке `ouroboros-auth` или cookie. Публичные пути: `/api/health`, `/auth/login`, `/auth/logout`.
7. Инструменты ограничены многослойной политикой: hardcoded sandbox реестра → `POLICY_SKIP`/`POLICY_CHECK_CONDITIONAL`/`POLICY_CHECK` → `claude_code_edit` revert guard поверх защищённой поверхности.
8. Запись в защищённую поверхность (BIBLE.md, safety.py, runtime_mode_policy.py, tools/registry.py, .github/workflows/ci.yml, Dockerfile) ограничена runtime-режимом; регистронезависимое сравнение путей.
9. Секреты в ответе `GET /api/settings` маскируются (`_SECRET_SETTING_KEYS`).
10. Inbound TLS/mTLS **отсутствует** (расчёт на loopback/LAN); outbound — HTTPS провайдерных SDK и git.

| OpenRouter capabilities timeout | 15000 мс | Хардкод |
| Claude runtime subprocess | 120000 мс | Настройка окружения |
| Local model health-check | 10000 мс | Проверка подпроцесса |
| max.request.size | не задан | Применяются дефолты Starlette (тело не ограничено) |

### 4.3. Надежность

1. Сервис рассчитан на режим 24/7 always-on (Kubernetes), один процесс.
2. Graceful shutdown реализован: lifespan finally — остановка A2A-сервера (≤5 с), локальной модели, воркеров (force=True), шины сообщений, затем git push (бюджет `Event.wait` 25 с).
3. Readiness probe — `GET /api/health` (всегда доступен, без auth, без проверки супервизора).
4. Liveness — `GET /api/state` (флаг `supervisor_ready`, `supervisor_error`, WS heartbeat-loop).
5. Replicas, Pod Disruption Budget, autoscaling, явный SLA/SLO — **не заданы** (см. 4.6).
6. Circuit breaker, request-rate-limiting, request-идемпотентность — **отсутствуют**.
7. Retry: A2A — нет (TimeoutError); git-sync bootstrap — без явного backoff; ClawHub — до 5 попыток, max sleep 60 с (хардкод).
8. Запись состояния атомарна (tmp→rename); ReviewState под exclusive-lock.

### 4.4. Мониторинг

#### 4.4.1. Стандартные метрики

Библиотека метрик (Prometheus/StatsD/Micrometer) **не используется**. Health/state-эндпоинты:

- `GET /api/health` — готовность (легковесный).
- `GET /api/state` — снапшот состояния (источник производных показателей).

#### 4.4.2. Кастомные бизнес-метрики

Метрики не экспортируются как time-series; значения доступны как поля `GET /api/state` (тип — Gauge/Counter по смыслу):

##### state.workers / queue

- **Тип**: Gauge — `workers_alive`, `workers_total`, `pending_count`, `running_count`.
- **Описание**: загрузка воркеров и очередь задач (из `supervisor/workers.py`, `supervisor/queue.py`).

##### state.budget

- **Тип**: Gauge — `spent_usd`, `budget_limit`, `budget_pct`.
- **Описание**: расход LLM против лимита `TOTAL_BUDGET` (из `supervisor/state.py`).

##### state.evolution_cycle

- **Тип**: Counter — `evolution_cycle`.
- **Описание**: число завершённых циклов эволюции.

#### 4.4.3. Вычисляемые метрики (Calculated Metrics)

| Категория | Метрика | Описание | Формула |
|-----------|---------|----------|---------|
| SLA | budget_pct | Доля израсходованного бюджета | `(spent_usd / budget_limit) * 100`, если `budget_limit > 0`, иначе 0 |
| SLA | queue_depth | Глубина очереди задач | `pending_count + running_count` |
| Бизнес-KPI | workers_utilization | Утилизация воркеров | `workers_alive / workers_total * 100` |

### 4.5. Аудит

1. События пишутся в `DATA_DIR/logs/events.jsonl`: `broadcast_partial_failure` (`ts, type, msg_type, dead_clients, total_clients`), `consciousness_tool_timeout` (`ts, type, tool_call_timeout, timeout_sec`).
2. Сообщения владельца — пер-задачно в `DATA_DIR/memory/owner_mailbox/{task_id}.jsonl` (`msg_id`, `ts`, `text`); пишет только инструмент `forward_to_worker`, супервизор не пишет.
3. Git-sync bootstrap/shutdown — в `server.log` (текстовый лог, не структурированный JSONL).
4. Единый журнал аудита доступа к `/api/settings` и неуспешных попыток аутентификации **отсутствует** (см. 4.6).

### 4.6. Известные ограничения и форк-эволюция

**Telegram (deprecated, в процессе удаления).** На 2026-05-18 Telegram-канал **фактически присутствует и функционирует** в коде: секрет `TELEGRAM_BOT_TOKEN` входит в маскируемые ключи, существует Telegram-мост, тогл `WEBUI_ONLY` скрывает Telegram-UI и блокирует мост. Это не противоречит разделу 2.8: мост существует сегодня и гейтится `WEBUI_ONLY`, и одновременно помечен как **deprecated и подлежащий полному удалению** (ещё не удалён). Новый код на Telegram-канал опираться не должен; после удаления раздел 2.8 (`WEBUI_ONLY`) и данный пункт подлежат актуализации.

**Playwright / browser-инструменты агента (удалены).** Интеграция агента с headless-браузером Playwright удалена полностью: инструменты `browse_page`, `browser_action`, производный анализ скриншотов (`analyze_screenshot`, `vlm_query`; модуль vision снят целиком) и алиасы `web_fetch`/`fetch` (реализованные поверх `browse_page`) больше не существуют; снята авто-установка Chromium в рантайме и thread-affinity-логика (`stateful_executor`, `cleanup_browser`). Обоснование — сокращение attack surface (произвольная навигация, исполнение JS через `browser_action action=evaluate`, произвольные исходящие HTTP/HTTPS), headless/k8s-окружение, вес и хрупкость бинарников Chromium. Замена не предусмотрена: агент теряет доступ к веб-контенту полностью; запрос удалённого инструмента обслуживается штатной обработкой неизвестного инструмента (правило 2.3.6), без fallback. `playwright` сохраняется **только** как test/dev-зависимость для UI/E2E-тестов веб-UI (сам веб-UI и его клиентский флоу не меняются). Env `PLAYWRIGHT_BROWSERS_PATH` снят из кода и `docs/ENV.md`. Продолжение линии упрощения runtime (ср. удаление PyInstaller/desktop-launcher pipeline).

**WEBUI_ONLY — пересмотр семантики.** Текущая семантика: тогл отключает все каналы коммуникации кроме Web UI (на практике — Telegram-мост). Планируется переопределение под появление проприетарных коннекторов как дополнительных каналов.

**Секреты.** Бизнес-инвариант «секреты не в env/конфигурации» (4.1.4) в текущем коде не выполняется: ключи/токены читаются из переменных окружения и `settings.json`, маскируются только в ответе `GET /api/settings`. Подключение корпоративного secret-менеджера — форк-доработка (не реализована).

**Корпоративный LLM-шлюз.** Слой `provider_models.py` — точка интеграции корп. LLM-прокси/шлюза; в текущем коде используются публичные провайдеры (Anthropic/OpenRouter/local). Конкретный корп. коннектор — форк-доработка.

**Стандартизация контрактов.** Нет OpenAPI; коды ошибок не стандартизованы (`{"error": string}` без `errorCode`); нет request-идемпотентности (`x-idempotency-key`); часть таймаутов/политик зашита в код (OpenRouter 15 с, ClawHub 5×/60 с, git-sync без retry).

**Надёжность в k8s.** Replicas, PDB, autoscaling, SLA/SLO в коде/Dockerfile не заданы (документы `specs/k8s-*.md` в данную спецификацию как факт не включены по решению владельца).

**Прочее.** A2A без request-dedup ключа (уникальность по `task_id` + TTL); WS msg_type расширяемы через runtime registry (extension-specific сообщения вне стабильного контракта); формат `OUROBOROS_TOOLS_ENABLED` (разделитель/экранирование) в коде не специфицирован; нет distributed tracing.
